# DirectStay PlatformLead Preview Build Playbook

Status: working playbook. Scope: DirectStay-level owner-acquisition preview sites, not Villa La Percha-specific.

## Purpose

A Preview Build should feel like DirectStay already understood the owner’s property, guests, market, and taste. It is a sales/onboarding artifact: fast, beautiful, personal, non-functional, and safe to share through a public-obscure preview URL.

The target reaction is not “nice template.” It is: “They saw what makes this place special.”

## Owner-trust sharing envelope

A Preview Build is not private merely because the URL is obscure. Treat it as owner-shareable but publicly reachable by anyone with the link.

Before sharing a Preview Build with an owner, record and include:

- Source basis: which owner-provided or public sources were used.
- Draft status: this is not live, not indexed, not linked from navigation, and not bookable.
- Assumption status: any inferred property/local/commercial claims must be labeled as draft assumptions.
- Correction path: owner can correct, remove, or replace any content before launch.
- Removal path: owner can ask DirectStay to take down or unshare the preview.
- Functionality status: inquiry, booking, payment, login, availability, and owner portal behavior are disabled unless launch-approved.

Do not share a Preview Build if it contains sensitive owner data, private contact details, unsupported safety/legal/accessibility claims, copied reviews without permission, or any claim that would embarrass DirectStay if the owner forwarded the URL to counsel.

## Required Preview Build packet

A Preview Build record is not a shareable Preview Build. Treat preview quality as two separate gates:

1. **Pre-render rubric preflight:** after the source audit, DESIGN brief, and content plan exist, score only the categories that can be judged before rendering. This can approve an internal render, but it cannot approve owner sharing.
2. **Post-render promotion gate:** after a real URL exists, rerun the full rubric against the rendered desktop and mobile experience, including page-order image inventory, technical polish, and safety constraints.

Before `READY_FOR_REVIEW` / owner-share review, it must include:

- Completed Property Photo + Geography Audit.
- Property-specific DESIGN.md/design brief.
- Preview content/section plan.
- Calendar/inquiry mock plan: a read-only date-selection module near the inquiry path showing how guests choose a stay window before submitting.
- Price comparison mock plan: an illustrative direct-vs-marketplace savings module, with owner-only calculation notes and clear assumption status.
- Local area guide plan: neighborhood/micro-destination details, things to do, restaurants/coffee, beach/trail/access notes, and owner-confirmation gaps.
- Fact register: observed facts, source URLs, and citations/notes.
- Assumption register: inferred claims and missing owner inputs.
- Owner-specific callouts, including at least one assumption/correction callout.
- Desktop QA evidence.
- Mobile QA evidence.
- Guest-copy QA checklist pass using `preview-build-guest-copy-qa-checklist.md`, with any internal strategy/process language moved out of guest sections.
- Rubric review with average score and safety pass/fail.
- Owner-share note explaining source basis, preview-only status, public-obscure/noindex status, correction path, and removal path.

If those are missing, the artifact is only a placeholder record — not a Preview Build suitable for Jaimal approval or owner sharing.

## Core rule

Design starts from the property’s evidence, not from a template.

Before choosing layout, palette, typography, or copy, build a Property Design Fingerprint from the owner’s submitted listing and any approved public sources.

## Creative review standard

Every serious Preview Build should target **50/50** on the creative review scorecard:

- visual distinctiveness;
- property/source fit;
- layout hierarchy;
- copy specificity;
- trust/conversion usefulness.

A score below 50 is not a failure, but it must produce a concrete improvement path. Do not treat 40+ as "done" if the missing points are fixable through better composition, sharper property-specific copy, stronger image sequencing, or a better archetype choice.

Before implementation hardens around one idea, run a three-archetype review:

1. Create or storyboard three materially different design directions, not minor palette swaps.
2. Score each direction against the five creative-review dimensions.
3. Pick the strongest production base.
4. Record what the winner should borrow from the losing variants so the safest version does not become visually bland.
5. If the winner is under 45/50, revise the concept before building or sharing it as a benchmark example.

Use this process for new Preview Builds and for old benchmarks before citing them as quality references. The goal is not merely safe previews; the goal is property-specific, owner-impressive preview sites that keep improving toward 50/50.

## Source intake

Use the owner-provided listing first. Supplement only with safe public sources when needed:

- Existing owner website
- Airbnb / VRBO / Booking listing
- Google/business profile
- Real estate page
- Public image gallery
- Public local/destination sources

Record source URLs and distinguish observed facts from inferred assumptions.

Never fabricate or imply unsupported amenities, reviews, ratings, distances, policies, pricing, taxes, availability, licensing, insurance, safety, accessibility, legal compliance, cancellation terms, booking/payment capability, or “best rate” claims. Do not copy OTA reviews, guest names, guest photos, platform ratings, or testimonial language unless permission/source rights are recorded and owner-approved.

## Property Design Fingerprint

For every Preview Build, create a short design brief capturing:

### Property basics

- Property name
- Location / destination context
- Property type: villa, cabin, condo, lake house, beach house, urban apartment, lodge, estate, boutique inn, etc.
- Guest capacity / bedroom count if public or owner-provided
- Known amenities
- Missing or uncertain facts

### Location-derived design cues

Use the exact destination itself as a palette and mood source. Avoid broad-category shortcuts.

Do not stop at “coastal,” “mountain,” or “city.” Go one level deeper:

- Chalk Sound / Turks and Caicos: electric turquoise lagoon, limestone/sand, white sun, salt-air brightness, tropical green, resort privacy.
- Cape Cod / New England coast: weathered cedar, dune grass, navy/cream, hydrangea, gray-blue water, classic restraint.
- Malibu / California coast: glass, bluff, pale concrete, golden hour, modern surf minimalism, canyon/desert edge.
- Greek islands: whitewash, cobalt, stone, bougainvillea, hard sunlight, stepped architecture.
- Lowcountry marsh: spartina grass, oyster shell, porch shade, tidal green-brown, live oak, lantern warmth.
- Mountain / forest: pine, stone, fog, warm wood, deep greens, softer shadows.
- Desert: clay, ochre, dusk gradients, sparse layouts, heat/sun contrast.
- Lake: freshwater blues, dock wood, sunset amber, calm horizontal compositions.
- Urban: sharper typography, editorial grids, neighborhood texture, night/day contrast.
- Countryside/farm: cream, earth, linen, garden greens, slower pacing.

Do not blindly use these defaults. Confirm against the actual listing photos and local context. If the photos contradict the geography stereotype, trust the photos.

### Photo-derived palette

Derive color direction from the actual images:

- Dominant exterior colors
- Water/sky/landscape tones
- Interior wall/floor/wood tones
- Accent colors already present in decor
- Texture notes: linen, rattan, stone, marble, tile, concrete, leather, painted wood, etc.

The preview should feel like it belongs to the photos, not like a separate SaaS skin.

### Owner/decorator style read

Infer the owner’s taste from the interior, carefully labeled as interpretation:

- Minimal / modern
- Coastal casual
- Traditional / classic
- Rustic / lodge
- Bohemian / collected
- Luxury resort / polished
- Family practical
- Historic / character-rich
- Design-forward / editorial

Use this to choose typography, section rhythm, copy tone, iconography, and visual density.

Examples:

- Minimal modern interiors → restrained palette, generous whitespace, crisp type, fewer decorative flourishes.
- Coastal casual interiors → airy layouts, natural textures, softer copy, sun/sand/water palette.
- Rustic lodge interiors → warmer palette, heavier typography, texture-forward sections, cozy practical copy.
- Design-forward interiors → editorial layout, bolder type contrast, artful image crops, fewer generic amenity cards.

### Guest promise

Summarize what guests are really buying:

- Quiet reset
- Family gathering
- Luxury privacy
- Adventure basecamp
- Romantic escape
- Beach week without friction
- Design-led city stay
- Local immersion

This becomes the hero concept.

### Competitive positioning

Explain why a guest should book this property direct:

- Better relationship with owner / host
- Clearer property-specific guidance
- Direct local knowledge
- Fewer marketplace distractions
- More confidence before inquiry
- Potentially better economics, without overpromising discounts

## Design selection process

1. Gather listing/source evidence, including the owner/manager-written description paragraphs — not just title, photos, amenity chips, and reviews.
2. Extract the source description’s property story, layout, guest rhythm, amenities, and tone into notes. Use it as inspiration for original DirectStay copy; do not copy/paste.
3. Audit photos before choosing design: exterior, views, interiors, materials, color temperature, decor, landscape, light, and repeated motifs.
   - Build a section image plan before rendering. The hero and first two content sections must use distinct images unless a written exception explains why the gallery is too thin.
   - Do not use the same or near-duplicate image for adjacent sections. If two images show the same room/angle, pick the stronger one and reserve the other only if the section needs that exact detail.
   - Inspect every selected image. Use only actual property photos, property views, or clearly relevant local-area photos. Reject promotional cards, review graphics, collages, screenshots, text overlays, OTA marketing banners, and images that are not truly the house/property/context.
   - If a section needs an image and no trustworthy source photo exists, leave the image out deliberately and mark the gap in the photo audit instead of filling it with a promotional/non-property asset.
4. Research or infer the specific geographical micro-context, not just the broad destination category.
5. Extract Property Design Fingerprint.
6. Run the Creative Review Standard: create or storyboard three materially different archetypes, score them against the 50-point scorecard, pick the strongest production base, and record what to borrow from the losing variants.
7. Choose a page archetype based on property and owner style, but let photos/geography override the archetype seed.
8. Choose palette from location + photos, with at least three palette notes tied directly to visible source evidence.
9. Create an override log: seed selected, what the seed contributes, what the property overrides, three design decisions only this property/location justifies, and one generic choice intentionally avoided.
10. Choose typography and layout rhythm from owner/decor style.
11. Select hero image and supporting image sequence for emotional flow. Confirm the sequence avoids repeated first-section imagery, avoids adjacent near-duplicates, and has no missing-image placeholders at the bottom of the page.
12. Draft guest-facing copy around the property’s specific promise.
13. Run the Copy Review Stack before treating rendered sections as more than a rough draft:
    - **Source-truth pass:** every sentence is either visible/source-backed, owner-confirmed, or explicitly framed as an assumption/correction item. This includes checking the actual owner/manager-written listing description paragraphs for story, layout, guest rhythm, and concrete details before drafting.
    - **VRBO-owner relevance pass:** remove anything that would not help a vacation-rental owner sell the stay, reduce guest friction, explain direct-booking value, or prepare onboarding.
    - **Guest usefulness pass:** replace strategy-memo language with concrete guest planning value: arrival, sleeping/gathering flow, beach/trail/town access, parking, pets, family logistics, weather/day plans, and inquiry confidence.
    - **Anti-AI voice pass:** delete brochure filler, “memorable property story,” “standalone brand,” “elevated experience,” “seamless,” “unlock,” “curated,” and any paragraph that could survive unchanged on another listing.
    - **Section-fit pass:** ask whether this belongs on the public guest page, an owner callout, the fact register, the assumption register, or nowhere. Move internal strategy copy out of guest sections.
14. Add the conversion trio before or near the inquiry section:
    - calendar mock (`calendarMock`) so date-aware inquiries feel central, not bolted on;
    - price comparison mock (`priceComparison`) so the owner sees how guest savings can be explained;
    - area guide (`areaGuide` or `locationGuide`) so the page sells the stay and destination, not just the house.
15. Add owner-view callouts explaining DirectStay strategy and assumptions/corrections.
16. Mark assumptions and missing owner inputs.
17. QA desktop/mobile and remove anything that feels generic.
    - During QA, run a rendered-image inventory in page order. Label each selected asset as property exterior, property interior, property amenity, local context, promotional/non-property, duplicate/near-duplicate, or unknown.
    - Rework before review if any guest-facing section uses promotional cards, review/rating graphics, text overlays, collages, OTA banners, unknown assets, or unrelated local filler as property imagery.
    - If the safe gallery is thin, prefer an intentional text-only section plus a recorded owner photo request over unsafe filler.
18. Record benchmark/rubric notes before considering the preview ready for Jaimal review.

## Thin-gallery benchmark standard

Some public owner sites expose useful source truth but not enough rights-safe property imagery to support a visual Preview Build. Treat these as constrained internal benchmarks, not failures.

Use a text-forward internal preview or content plan when all of these are true:

- the source gives enough property facts, owner voice, or guest-planning detail to learn from;
- the inspected image set has zero or one trustworthy property-specific image;
- exterior, amenity, local, or attraction images cannot be tied safely to the property;
- owner-share would require a full owner/gallery photo set before hero and section art can be chosen.

For these cases:

- keep hero and amenity sections text-forward until owner/gallery images exist;
- use the single confirmed property image only where it proves what it shows, usually interior or owner-care trust;
- mark river, dock, pool, spa, beach, wildlife, restaurant, distance, parking, pet, accessibility, and safety details as owner-confirmation-needed unless directly sourced;
- write the conversion trio as guest-planning questions and structure-only clarity, not as proof of live booking, rates, discounts, or amenities;
- score the artifact as a learning benchmark or builder-ready plan, not as an owner-share candidate.

## Benchmark promotion gate

Internal benchmark previews are allowed to start rough, but they do not become examples for future owner work until the rework evidence proves they meet the current bar. Before citing a benchmark as a quality reference, copying its structure into a seed, or using it in an approval packet, complete this promotion gate:

1. **Rendered-page evidence:** capture or inspect the live guest page after the latest copy/image changes, not only the stored artifact data.
2. **Image inventory in page order:** list hero and each rendered section image; label every image as property exterior, property interior, property amenity, property view, local context, text-only, rejected/promotional, duplicate/near-duplicate, or unknown.
3. **First-screen distinction:** confirm the hero and first two visible content sections do not repeat the same room/view/angle unless the thin-gallery exception is documented.
4. **Bad-copy scan:** search guest-visible copy for benchmark/process language such as `benchmark`, `source description`, `source material`, `final live copy`, `preview uses mock`, `strategy`, and any city/property badges left over from another preview.
5. **Module fit check:** verify calendar, price, and area-guide modules read like guest planning help for this exact stay type, with DirectStay/process rationale kept in owner-only notes or artifacts.
6. **Owner-share blockers:** record remaining blockers separately from the benchmark score, especially photo rights, exact rates/fees/taxes, availability, occupancy, parking/access, rules, safety claims, local recommendations, and cancellation/payment terms.
7. **Score after rework:** score the promoted version, not the first generated version. If visual fit, copy quality, mobile composition, or technical polish stays below 4, keep it as a learning artifact rather than a model example.

Promotion verdicts should use plain labels: `learning artifact`, `Jaimal/internal review example`, or `owner-share candidate after Jaimal approval`. A benchmark is never owner-shareable just because it scored well; the owner-share envelope and approval gate still apply.

## Conversion trio: required modules

Every serious Preview Build should include these three modules unless there is a recorded reason not to. They are not live functionality; they are a sales/process mock that shows how a DirectStay site can outperform an OTA listing.

Recent benchmarks show the trio fails when it sounds like product strategy instead of guest planning. Write these sections in plain rental-site language first, then put DirectStay/process rationale in owner-view notes.

For family/group lake houses, the conversion trio should lead with planning confidence: sleeping layout, party size, children, dock/boat/PWC/kayak/hot-tub expectations, parking, quiet hours, and whether the stay is a quiet family trip or a celebration. Avoid generic “luxury lakefront” copy until the practical decision path is clear.

For Gulf Coast river/canal retreats, use the `design-seeds/gulf-river-retreat.DESIGN.md` seed as a starting point only after source and photo evidence support it. Lead with private waterway rhythm, owner care, pool/spa/dock/kayak rule confirmation, and Sarasota/Gulf local-guide questions. Do not let a riverfront home collapse into generic Florida beach-villa copy or unsupported wildlife/distance claims.

### Calendar / inquiry context

Place a read-only calendar or date-window mock near the inquiry path. The goal is to show that guests can arrive with dates, party size, and stay context before writing a message.

Tailor the date prompt to the property type: weekend/event windows for urban stays, weekly or holiday blocks for beach/lake houses, seasonal long weekends for cabins, and flexible arrival questions for complex villas or estates.

Owner note should explain:

- what would power the live calendar: owner-approved availability, holds, minimum stay rules, and seasonal rules;
- that final booking/payment is still disabled until launch readiness;
- how date-aware inquiries reduce back-and-forth.

Never imply real availability in a Preview Build unless it is sourced from owner-approved availability data and labeled appropriately.

### Price comparison / savings story

Add an illustrative direct-vs-marketplace comparison. The module should show how a guest might understand savings, but it must remain clearly mock/assumption-based until rates, taxes, cleaning fees, platform/service-fee assumptions, and owner terms are approved.

Guest-facing copy should be about clarity, fewer surprise fees, and a direct conversation with the host/owner — not “we beat marketplaces.” Owner-view notes can explain the commercial strategy and assumptions.

Owner note should show calculation structure:

```md
Direct estimate = nightly rate × nights + cleaning/owner fees + taxes
Marketplace-style estimate = same lodging basis + estimated platform/service fee + marketplace fee assumptions
Potential guest savings = marketplace-style total - direct estimate
```

Guardrails:

- Do not say “best rate,” “guaranteed savings,” or “lowest price” unless approved data proves it.
- Keep exact numbers illustrative or source-backed.
- Put calculation assumptions in owner view, not guest view, until approved.

### Area guide / micro-destination proof

Add page details about the property’s actual area: things to do, restaurants/coffee, beach/trail/access notes, rainy-day options, family/pet/romantic/adventure guidance as relevant.

The goal is to demonstrate DirectStay’s local-intelligence advantage. Generic “near beaches and restaurants” is not enough. Do not overclaim exact walk times or recommendations just to sound local; use owner-confirmation placeholders when the source material is thin.

Fact/assumption handling:

- Facts visible from public sources can be used with notes.
- Exact distances, walking times, pet rules, venue recommendations, and seasonal conditions should be owner-confirmed unless sourced from reliable public pages.
- If the location is inferred from listing text/photos, label the local guide as draft and confirmation-needed.

## Page archetypes

Use archetypes as starting points, not rigid templates. The archetype must always be subordinate to the actual property photos and specific geography. Two coastal villas should look different if one is in Chalk Sound, one is in Cape Cod, one is in Malibu, one is in the Greek islands, and one is on a Lowcountry marsh. “Coastal” is a broad category, not a design direction.

### Luxury villa / resort alternative

- Editorial hero
- Big scenic image sequencing
- Elevated amenities and privacy language
- Local guide / concierge confidence
- Palette often drawn from water, stone, linens, sunset, foliage

### Cozy cabin / nature retreat

- Warm, tactile layout
- Story-led sections
- Fireplace/deck/view moments
- Practical adventure/local notes
- Palette from wood, pine, stone, fog, firelight

### Family beach / lake house

- Clear sleeping/amenity information
- Relaxed but polished visuals
- Strong “easy week together” promise
- Local activity cards
- Palette from sand/water/dock/sunset/interior textiles

### Modern design stay

- Crisp editorial structure
- Strong typography
- Fewer, larger image moments
- Architecture/interior-design language
- Palette from materials: concrete, oak, marble, black metal, white walls, accent objects

### Urban / neighborhood stay

- Neighborhood-forward positioning
- Map/local texture
- Sharper grid
- Food/walkability/culture copy
- Palette from city materials, night lights, interiors, local landmarks

### Boutique multi-unit / small hospitality brand

- Brand-level story
- Unit comparison
- Shared amenities
- Trust/operations cues
- Palette from logo/architecture/location, but avoid corporate blandness

## Preview page sections

Default section set:

1. Hero: property-specific emotional promise.
2. Owner-view strategy callout: why this positioning works.
3. Signature stay moments: 3–5 specific reasons guests choose it.
4. Image-led story section: exterior/interior/location flow.
5. Sleeping / amenities / practical confidence.
6. Location guide teaser: nearby experiences and what guests can expect.
7. Direct-booking value: owner relationship, clarity, local knowledge, preview-only inquiry path.
8. Preview-only CTA: non-functional, clearly marked.
9. Missing inputs: what DirectStay would refine with the owner.

## Owner-view callouts

Owner callouts should sell the thought process without ruining the beauty of the page.

Good callouts explain:

- Why the hero leads with a guest emotion, not just bedroom count.
- How image order creates desire and confidence.
- How local-guide content creates SEO value and reduces pre-booking friction.
- How direct inquiries preserve relationship and context.
- What assumptions will be replaced with owner truth during onboarding.

Avoid generic platform boasting.

## Anti-template checklist

Before sharing a Preview Build, confirm:

- The Copy Review Stack has been run and recorded in the rubric notes.
- Guest-facing copy reads like a vacation-rental site, not an AI strategy memo.
- Internal DirectStay/process logic lives in owner callouts or artifacts, not in public guest paragraphs.
- Any sentence that would feel odd on a VRBO/Airbnb owner’s direct-booking page has been removed or moved to owner-only notes.
- The color palette clearly relates to the property photos/location.
- The typography/layout fit the owner’s apparent decor style.
- The hero could not be pasted onto a random rental without sounding wrong.
- At least 5 copy details are property/location-specific.
- Image crops/order tell a coherent guest story.
- Owner callouts are specific to this property’s opportunity.
- No Villa La Percha references are present unless intentionally approved.
- No fake reviews, fake distances, fake amenities, fake availability, or fake pricing.
- Inquiry/payment/booking functionality is disabled or preview-only.
- Page is noindex/public-obscure and not in nav/sitemap.
- Mobile hero and image flow are intentionally composed.

## Build quality bar

Preview Builds should be fast, but not cheap. If the lead sent a listing, we should be able to infer enough from their photos/location/decor to make the first draft feel personal. If source material is too thin, say so and make the missing inputs part of the preview narrative rather than inventing polish.
